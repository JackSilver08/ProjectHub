CHƯƠNG 1: TỔNG QUAN DỰ ÁN
1.1. BỐI CẢNH VÀ MỤC TIÊU
1.1.1. Bối cảnh
Hiện tại, quy trình quản lý công việc và dự án nội bộ đang gặp một số vấn đề:
Thông tin phân tán: Trao đổi công việc diễn ra trên nhiều kênh (Zalo, Email, Excel) khiến việc tra cứu lịch sử và ngữ cảnh rất khó khăn.
Thiếu tính minh bạch: Khó theo dõi tiến độ tổng thể của một dự án hoặc khối lượng công việc của từng cá nhân.
Giao diện phức tạp: Các phần mềm quản lý dự án chuyên sâu (như Jira, MS Project) thường quá phức tạp và cần thời gian đào tạo lâu, tạo rào cản cho nhân viên không chuyên kỹ thuật.

1.1.2. Mục tiêu Dự án

Xây dựng hệ thống ProjectHub với các tiêu chí cốt lõi:
Giao diện quen thuộc: Lấy cảm hứng từ mô hình Sub/Thread để tạo môi trường làm việc thân thiện, tập trung vào thảo luận và sự minh bạch.
Quy trình: Áp dụng quy trình trạng thái công việc (Workflow) trong quản lý.
Tương tác thời gian thực: Giảm thiểu độ trễ thông tin thông qua hệ thống thông báo và bình luận tức thì.
Lưu trữ tập trung: Toàn bộ tài liệu, file đính kèm và lịch sử thảo luận đều gắn liền với từng đầu việc cụ thể.

1.2. PHẠM VI DỰ ÁN
1.2.1. Các tính năng chính
Hệ thống bao gồm các phân hệ (modules) chức năng sau:
Phân hệ Quản trị (Admin & Identity):
Đăng nhập, Đăng ký (hoặc Import user), Quản lý hồ sơ nhân viên.
Phân quyền theo vai trò: Administrator, Manager, Member.
Phân hệ Dự án (Projects/Subs):
Tạo mới, chỉnh sửa thông tin dự án.
Module Wiki: Soạn thảo tài liệu hướng dẫn, quy định nội bộ cho từng dự án.
Quản lý thành viên và nhãn (Labels/Tags) riêng cho dự án.
Phân hệ Công việc (Tasks/Threads):
Tạo công việc với đầy đủ thuộc tính: Tiêu đề, Mô tả (Rich Text), Deadline, Độ ưu tiên, Người thực hiện.
Private Task: Cơ chế bảo mật cho các công việc nhạy cảm (chỉ người liên quan mới thấy nội dung).
Workflow: Chuyển đổi trạng thái (New -> In Progress -> Review -> Done -> Cancelled).
Phân hệ Tương tác:
Bình luận đa cấp (Nested Comments).
Nhắc tên thành viên (@Mention) và gửi thông báo.
Đính kèm tập tin vào Task và Comment.
Hệ thống Voting (Upvote/Downvote) để thể hiện sự đồng thuận.
Phân hệ Báo cáo & Tiện ích:
Dashboard thống kê tiến độ dự án (%).
Tìm kiếm toàn văn (Full-text search).
Nhật ký hoạt động (Audit Logs).

1.3. TỔNG QUAN VỀ CÔNG NGHỆ
Dự án sử dụng nền tảng công nghệ Microsoft .NET mới nhất để đảm bảo hiệu năng và khả năng bảo trì dài hạn.

1.3.1. Kiến trúc Tổng thể
Mô hình: Modular Monolith (Một khối phân chia theo module nghiệp vụ).
Lý do lựa chọn: Giảm độ phức tạp khi triển khai, dễ dàng debug và phát triển nhanh, phù hợp với quy mô team và user hiện tại.
1.3.2. Ngôn ngữ & Framework (Backend)
Ngôn ngữ: C#.
Framework: ASP.NET Core 10.
ORM: Entity Framework Core (Code-First approach).
Real-time: SignalR Core (Dùng cho thông báo và cập nhật Comment).
Authentication: ASP.NET Core Identity (Cookie-based auth).
1.3.3. Cơ sở dữ liệu (Database)
Database Engine: PostgreSQL 16
Lý do lựa chọn: Mã nguồn mở, hiệu năng cao, hỗ trợ tốt JSONB (cho Audit Log) và Full-text Search
File Storage: Lưu trữ vật lý trên Server thông qua PhysicalFileProvider, quản lý đường dẫn trong Database.
1.3.4. Giao diện người dùng (Frontend)
Server-side Rendering (SSR): Razor Pages. Đảm nhiệm việc render khung trang, routing, SEO và bảo mật phân quyền ban đầu.
Client-side Interactivity: Vue.js 3.
Vue.js sẽ được nhúng vào các vùng cần tương tác cao như: bình luận, Form chọn người (User Picker), Danh sách thông báo.
Các trang tĩnh (About, Wiki view) chỉ dùng HTML/CSS thuần để tối ưu tốc độ.
CSS Framework: Bootstrap 5 (tùy chỉnh theme màu sắc theo Brand công ty).
1.3.5. Môi trường triển khai (Deployment Environment)
OS: Windows Server (IIS) hoặc Linux (Ubuntu + Nginx/Kestrel).

2.3. MODULE QUẢN LÝ CÔNG VIỆC
Đây là thực thể chính của hệ thống, nơi mọi hoạt động và báo cáo diễn ra.

2.3.1. Cấu trúc thông tin Task

Mỗi Task bao gồm các trường thông tin sau:
Tiêu đề: Ngắn gọn, súc tích.
Mô tả chi tiết: Hỗ trợ Rich Text (In đậm, nghiêng, list, code block).
Người thực hiện (Assignee): Một hoặc nhiều thành viên chịu trách nhiệm chính.
Thời hạn (Deadline): Ngày/Giờ phải hoàn thành.
Độ ưu tiên (Priority):
Low: Làm khi rảnh.
Medium: Bình thường (Mặc định).
High: Cần làm sớm.
Critical: Khẩn cấp, block tiến độ dự án.
Tính tiến độ:
True: Task này là công việc thực tế (VD: Design UI, Code API), khi hoàn thành sẽ tăng % dự án.
False: Task này là thảo luận, hành chính, hoặc support (VD: Họp team, Brainstorm), hoàn thành hay không không ảnh hưởng % dự án.
Chế độ riêng tư (Private Mode): Đánh dấu task nhạy cảm.
2.3.2. Quy trình xử lý (Workflow)
Hệ thống áp dụng Quy trình trạng thái, người dùng không thể tự định nghĩa trạng thái mới ngoài luồng này để đảm bảo thống nhất toàn công ty.
Các trạng thái (States):
New: Mới khởi tạo, chưa ai làm.
In Progress: Assignee đã nhận việc và đang thực hiện.
On Hold: Tạm dừng do thiếu tài nguyên hoặc bị Block bởi task khác.
In Review: Đã làm xong, chờ Manager hoặc Tester kiểm tra.
Done: Đã hoàn thành và được chấp nhận.
Cancelled: Hủy bỏ, không làm nữa.
Quy tắc chuyển đổi:
Member: Chỉ được chuyển: New -> In Progress -> In Review. Không được tự ý chuyển sang Done (nếu quy trình yêu cầu duyệt) hoặc Cancelled.
Manager: Có quyền chuyển sang bất kỳ trạng thái nào (bao gồm Done và Cancelled).
2.3.3. Cơ chế Bảo mật
Đây là tính năng quan trọng để bảo vệ thông tin nhạy cảm (Lương thưởng, Sa thải, Dự án bí mật) nhưng vẫn giữ tính minh bạch về khối lượng công việc.

Quy tắc hiển thị:
Người có quyền (Manager, Creator, Assignee): Nhìn thấy toàn bộ thông tin như Task bình thường.
Người không có quyền (Member khác, Viewer):
Trên danh sách: Vẫn thấy Task tồn tại.
Tiêu đề: Bị ẩn, thay thế bằng văn bản hệ thống: "Restricted Task #[ID]".
Nội dung/File: Ẩn hoàn toàn.
Comment: Ẩn hoàn toàn.
Trạng thái/Deadline: Vẫn hiển thị (để biết member kia đang bận hay rảnh).
2.3.4. Logic Tính Tiến độ Dự án
Tiến độ (%) của một Dự án (Sub) được tính tự động dựa trên các Task có cờ Contributes to Progress = True.
Công thức:



Lưu ý: Các Task trạng thái Cancelled sẽ bị loại khỏi mẫu số để không làm sai lệch báo cáo.

2.4. MODULE TƯƠNG TÁC & THẢO LUẬN
Mô phỏng trải nghiệm mạng xã hội để khuyến khích trao đổi.
2.4.1. Bình luận & Báo cáo (Comments)
Cấu trúc: Cây bình luận đa cấp (Nested Comments/Reply).
Chức năng:
Soạn thảo Markdown.
Mention (@username): Gõ @ sẽ hiện danh sách thành viên dự án để chọn. Người được nhắc sẽ nhận thông báo.
Báo cáo: Vì không có Sub-task, thành viên phải comment để cập nhật tiến độ nhỏ (VD: "Đã xong phần API, đang làm UI").
2.4.2. Hệ thống Vote (Voting)
Cơ chế: Nút Upvote (Mũi tên lên) và Downvote (Mũi tên xuống) tại Task và Comment.
Mục đích:
Dùng để biểu quyết nhanh (VD: Manager post một giải pháp, team vote Up nếu đồng ý).
Thể hiện sự tín nhiệm cho một câu trả lời/giải pháp tốt.
Quy tắc: Số lượng Vote KHÔNG ảnh hưởng đến thứ tự sắp xếp (Sorting) của Task trên Dashboard (để tránh việc Task quan trọng nhưng ít người quan tâm bị trôi xuống dưới).
2.4.3. Quản lý File đính kèm (Attachments)
File đính kèm được phân loại rõ ràng theo ngữ cảnh:
Project Files: Tài liệu chung (Nằm ở Sidebar dự án).
Task Attachments: File đề bài, mockup (Nằm ngay dưới nội dung Task).
Comment Attachments: File kết quả công việc, file log lỗi (Nằm trong từng comment).

2.5. CÁC TÍNH NĂNG HỖ TRỢ 
2.5.1. Trung tâm Thông báo (Notification Center)
Hệ thống gửi thông báo qua 2 kênh: In-App (Quả chuông) và Email.
Các sự kiện kích hoạt (Triggers):
Assigned: Khi người dùng được gán vào một Task.
Mentioned: Khi người dùng được nhắc tên (@) trong Comment/Task.
Deadline Approaching: Cảnh báo trước 24h khi Task sắp hết hạn.
Status Changed: Khi Task mình đang theo dõi chuyển trạng thái (VD: Từ In Review sang Done).
2.5.2. Nhật ký Thay đổi (Audit Log)
Ghi lại lịch sử chỉnh sửa để quy trách nhiệm. Log được hiển thị ở tab "History" trong chi tiết Task.

Các dữ liệu cần ghi:
Thay đổi Tiêu đề/Nội dung Task.
Thay đổi Trạng thái (VD: Ông A chuyển từ Doing -> Done).
Thay đổi Deadline (Rất quan trọng).
Thay đổi Người thực hiện (Assignee).
2.5.3. Tìm kiếm & Lọc (Search & Filter)
Bộ lọc (Filter):
Theo trạng thái (New, Doing, Done).
Theo Assignee (Tasks của tôi, Tasks của A).
Theo Tags (Chỉ hiện Task "Bug" hoặc "Urgent").
Sắp xếp (Sort):
Newest: Mới tạo nhất.
Deadline: Sắp hết hạn nhất (Ưu tiên hiển thị).
Ghim (Pin): Các Task được Manager ghim sẽ luôn nằm trên cùng bất kể bộ lọc.

CHƯƠNG 3: THIẾT KẾ HỆ THỐNG
Chương này mô tả chi tiết kiến trúc kỹ thuật của ProjectHub, bao gồm sơ đồ hệ thống, cấu trúc mã nguồn và thiết kế cơ sở dữ liệu.

3.1. KIẾN TRÚC TỔNG THỂ

Hệ thống được thiết kế theo mô hình Modular Monolith trên nền tảng .NET 10. Đây là mô hình cân bằng tốt nhất giữa sự đơn giản khi triển khai và khả năng mở rộng trong tương lai.

3.1.1. Sơ đồ Kiến trúc Hệ thống

Hệ thống hoạt động theo mô hình Client-Server tiêu chuẩn. Dưới đây là sơ đồ minh họa luồng tương tác giữa các thành phần:


Giải thích luồng dữ liệu:
Client Layer:
Giao diện tĩnh: Được render bởi Razor Pages (HTML/CSS).
Giao diện động: Các vùng tương tác cao (Comment, Notification) sử dụng Vue.js gọi API ngầm.
Real-time: Kết nối WebSocket qua SignalR để nhận thông báo tức thì.
Application Layer:
Chạy trên ASP.NET Core 10.
Xử lý Routing, Authentication, Authorization.
Chứa toàn bộ Business Logic (Quy trình Task, Tính toán tiến độ).
Data Layer:
Database: PostgreSQL 16+ chứa dữ liệu có cấu trúc (Users, Projects, Tasks).
File Storage: Hệ thống tệp tin vật lý để lưu trữ file đính kèm (lưu vào /wwwroot/uploads).
3.1.2. Mô hình Phân lớp (Layered Architecture)
Dù là Monolith, mã nguồn vẫn được chia tách logic thành các lớp để đảm bảo Clean Code. Sơ đồ dưới đây thể hiện sự phụ thuộc giữa các lớp:


Presentation Layer:

3.2. THIẾT KẾ CƠ SỞ DỮ LIỆU
Hệ thống sử dụng PostgreSQL làm cơ sở dữ liệu chính. Việc thiết kế tuân thủ các nguyên tắc chuẩn hóa (Normalization) đến dạng chuẩn 3NF, đồng thời phi chuẩn hóa (Denormalization) một số trường hợp cụ thể (như tính toán tiến độ, cache tên user) để tối ưu hiệu năng đọc.

3.2.1. Quy ước đặt tên & Kiểu dữ liệu
Bảng (Table): Sử dụng snake_case số nhiều. Ví dụ: users, projects, task_comments.
Cột (Column): Sử dụng snake_case. Ví dụ: created_at, is_active.
Khóa chính (PK): Sử dụng UUID (v4) thay vì Integer tự tăng.
Lý do: Tăng tính bảo mật, dễ dàng merge dữ liệu từ nhiều nguồn, phù hợp với kiến trúc phân tán nếu cần mở rộng sau này.
Thời gian: Sử dụng TIMESTAMPTZ (Timestamp with Time Zone) để đảm bảo đồng nhất múi giờ khi triển khai Server ở các vị trí địa lý khác nhau.

3.2.4. Đánh chỉ mục (Indexing)
Để đảm bảo truy vấn nhanh khi dữ liệu lớn, các chỉ mục sau BẮT BUỘC phải được tạo:
Index tìm kiếm Task: CREATE INDEX idx_tasks_project_status ON tasks (project_id, status); -> Tối ưu cho việc lọc task trong Dashboard.
Index thời hạn: CREATE INDEX idx_tasks_deadline ON tasks (deadline) WHERE status != 4 AND status != 5; -> Tối ưu cho job quét deadline (chỉ quét task chưa xong).
Index Assignee: CREATE INDEX idx_tasks_assignee ON tasks (assignee_id); -> Để xem "Task của tôi".
Index Comment: CREATE INDEX idx_comments_task ON comments (task_id, created_at); -> Để load comment theo thứ tự thời gian.
Full-text Search: Tạo GIN Index cho cột title và description của bảng Tasks để hỗ trợ tìm kiếm từ khóa.

Xấu: "Fix code", "Update"

Từ nhánh feature/... vào develop.
Mô tả rõ: Làm chức năng gì? Test như thế nào?
Code Review:
Bắt buộc ít nhất 1 người review trước khi Merge.

Merge: Sử dụng chế độ "Squash and Merge" (nếu cần gọn history) hoặc "Merge Commit".

SDK: .NET SDK.
Database: PostgreSQL 16+.
Tools:
pgAdmin 4 (hoặc tương đương): Để quản lý giao diện Database.
Git SCM: Để quản lý mã nguồn.
Node.js (Optional): Chỉ cần nếu muốn build lại CSS từ SASS hoặc quản lý thư viện JS (tuy nhiên dự án này ưu tiên dùng CDN/LibMan cho đơn giản).
6.1.2. Các bước cài đặt 
Bước 1: Lấy mã nguồn
git clone project-url

Bước 2: Cấu hình kết nối
Copy file appsettings.json thành appsettings.Development.json.
Cập nhật chuỗi kết nối (Connection String) phù hợp với máy cá nhân:
"ConnectionStrings": {
  "DefaultConnection": "Host=localhost; Port=5432; Database=DBNAME; Username=postgres; Password=yourPassword"
}

Bước 3: Khởi tạo Cơ sở dữ liệu
Dự án sử dụng Entity Framework Core Code-First. Chạy lệnh sau tại thư mục gốc của Project để sinh Database và các bảng:

# Cài đặt công cụ EF (nếu chưa có)
dotnet tool install --global dotnet-ef

# Chạy Migration
dotnet ef database update

6.2. KẾ HOẠCH SAO LƯU & PHỤC HỒI (BACKUP & RECOVERY)
Dữ liệu là tài sản quan trọng nhất. Vì hệ thống lưu trữ File vật lý và Database riêng biệt, cần backup cả hai thành phần này.

6.2.1. Sao lưu Cơ sở dữ liệu (PostgreSQL)
Sử dụng công cụ pg_dump để xuất dữ liệu ra file SQL.

6.2.2. Sao lưu File đính kèm (File Storage)
Toàn bộ file upload nằm trong thư mục wwwroot/uploads.
Phương pháp: Sử dụng Robocopy (Windows) hoặc Rsync (Linux) để đồng bộ sang ổ cứng khác hoặc Cloud Storage (AWS S3 / Google Drive).
Script mẫu:
robocopy "C:\Inetpub\wwwroot\ProjectHub\wwwroot\uploads" "D:\Backups\Files" /MIR /R:3 /W:10

6.2.3. Quy trình Phục hồi (Recovery)
Khi có sự cố sập server hoặc mất dữ liệu:
Dừng Website (IIS/Kestrel).
Restore Database:
psql -U postgres -d ProjectHub_Prod -f backup_file.sql
Restore Files: Copy ngược thư mục backup vào wwwroot/uploads.
Khởi động lại Website.
